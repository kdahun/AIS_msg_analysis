# 분석 프로그램 수정 계획 (수집 장소 · FSR 반영)

DB에 **수집 장소(`rx_sites`, `site_id`)**와 **FSR(`ais_fsr`)**이 들어오면서 대시보드가 손봐야 할 것들을 정리한다.

| 갈래 | 성격 | 안 하면 |
|---|---|---|
| **A. 구간(segment) 도입** | 지금 코드가 틀린 값을 내는 것 | 거리·보고주기가 오염, 오탐 1,369건 |
| **B. FSR 활용** | 새로 할 수 있게 된 것 | 추정에 의존한 채로 남음 |
| **C. 장비 상태 표시** | 현상을 볼 수 없는 것 | 이상 구간을 모르고 결과만 봄 |

A가 나머지의 전제다.

---

## 0. 분석의 계층

```
대분류   장소            좌표·전파환경이 다름. baseline 이 갈림
중분류   연속 수신 구간    장비가 계속 켜져 있던 구간
필터     날짜            화면에서 걸러 보기만. 계산을 자르지 않음
```

**"자른다"의 의미**: 선박별 시계열(보고 간격·슬롯 체인)을 계산할 때 그 경계를 넘어 다음 메시지를 찾지 않는다는 뜻. 코드로는 `groupby`에 넣느냐의 문제다.

```python
df.groupby("mmsi")                     # 지금 — 장소·중단을 넘나듦
df.groupby(["segment_id", "mmsi"])     # 바꿀 것 — 구간 안에서만
```

**날짜는 넣지 않는다.** 자정에는 수신기도 선박도 아무 변화가 없다. 23:59:55에 보고한 배가 00:00:05에 또 보고하면 정상 10초 간격이다. 여기서 끊으면 멀쩡한 간격 **362개**를 버린다(부산역 수집이 자정을 끊김 없이 통과함).

계산은 구간 전체로 끝내고, 화면에 뿌릴 때만 날짜로 거른다. 그래서 날짜를 바꿔도 재계산이 없다.

## A. 구간(segment) 도입

### A-1. 구간의 정의

**장소가 바뀌거나, 전 선박 무수신이 5초 이상이면 새 구간**으로 나눈다.

임계 5초의 근거 — 정상 상태에서 스트림이 조용한 시간의 분포:

| 무수신 시간 | busan_station | kmou |
|---|---|---|
| 0.5초 미만 | 817,899개 (99.98%) | 192,032개 (99.71%) |
| 0.5~1초 | 146개 | 549개 |
| 1~2초 | 0개 | 3개 |
| **2~5초** | **0개** | **0개** |
| 실제 중단 | 17초 | 95초, 105초, 145초 |

정상일 때 2초 이상 조용한 적이 한 번도 없고 실제 중단은 최소 17초다. 그 사이가 비어 있어 임계를 어디로 잡아도 되며, 5초면 정상 최대의 2.5배이자 최소 중단의 1/3이라 안전하다.

구간은 장소가 바뀌어도 끊기므로 **항상 한 장소 안에 완전히 들어간다.** 같은 장소를 나중에 재방문해도 그 사이 공백 때문에 자동으로 새 구간이 된다.

### A-2. 이번 데이터의 구간 6개

| 구간 | 장소 | 시작 ~ 끝 (KST) | 앞의 경계 |
|---|---|---|---|
| 1 | 부산역 | 06-09 17:56:42 ~ 18:13:56 | 수집 시작 |
| 2 | 부산역 | 06-09 18:14:14 ~ 06-10 08:30:15 | 17초 중단 *(자정 통과)* |
| 3 | 해양대 | 06-10 10:33:23 ~ 10:50:42 | **장소 이동 (123분)** |
| 4 | 해양대 | 10:52:18 ~ 11:00:00 | 96초 중단 |
| 5 | 해양대 | 11:01:45 ~ 11:21:20 | 105초 중단 |
| 6 | 해양대 | 11:23:45 ~ 15:19:59 | 145초 중단 |

### A-3. 없어지는 오탐

```
전체 보고 간격            1,008,294개
  장소 이동을 넘는 간격        317개  ← 지금 위반으로 오탐
  수집 중단을 넘는 간격      1,052개  ← 지금 위반으로 오탐
  ─────────────────────────────────
  합계                    1,369개  (0.14%)
```

짧은 중단(17·96·105·145초)이 장소 이동보다 오탐을 더 많이 만든다. 이동 때는 2시간이 비어 양쪽에 다 잡힌 선박이 적지만, 짧은 중단은 대부분의 선박이 양쪽에 걸치기 때문이다.

구간 경계의 마지막 메시지는 "다음 보고"가 없으므로 위반이 아니라 **검증 보류**로 둔다. 기존 `RI_LOST_*` / `SLOT_UNVERIF_*` 체계에 그대로 얹힌다.

| 위치 | 현재 | 수정 |
|---|---|---|

| [logic.py](../dashboard/tabs/msg_analysis/logic.py) 슬롯체인·유실·침범 | 전체 대상 | 구간 안으로 한정 |

**거리가 틀리는 규모**: 두 장소가 5.6km 떨어져 있다. 부산역 구간 84만 건이 해양대 기준으로 계산된다.

**잡음층에 채널을 넣는 이유**: FSR 실측에서 채널 A와 B의 잡음이 다르다. 지금은 A/B를 섞어 하나의 중앙값을 낸다.

### A-5. `v_vsi` 뷰와 나머지 탭

`v_vsi`에 `site_id`가 없어 RSSI 탭 3개([by_mmsi](../dashboard/tabs/rssi_snr_by_mmsi.py) / [by_time](../dashboard/tabs/rssi_snr_by_time.py) / [by_message](../dashboard/tabs/rssi_snr_by_message.py))가 두 장소를 합산한다.

- `v_vsi`에 `site_id` 추가 (`vsi_slot`은 완료)
- 사이드바에 **장소 / 구간 / 날짜** 선택. 앞의 둘은 계산 단위, 날짜는 표시 필터
- "전체"일 때도 RSSI 계열은 장소별로 색을 나눈다. 두 지점의 RSSI 절대값은 직접 비교 대상이 아니다

### A-6. `UNIV_START` 제거와 장소별 baseline

[queries.py](../dashboard/core/queries.py) `load_dynamic_positions`의 `recv_time >= UNIV_START` 필터는 "앞 구간 좌표를 몰라서" 넣은 것이다. 삭제하면 **신호 유효성 분석 대상이 4배**로 늘어난다.

단 [signal_validity.py](../dashboard/tabs/signal_validity.py)의 거리–RSSI baseline은 **장소별로 따로** 만든다. 안테나 높이·주변 지형이 달라 섞으면 오염된다.

## B. FSR 활용

### B-1. 잡음층: 추정 → 실측 (최우선 검증)

지금 [data.py](../dashboard/tabs/msg_analysis/data.py)는 잡음층을 `median(RSSI − SNR)`로 **추정**한다. FSR은 수신기가 **측정한** 값을 분·채널 단위로 준다.

원문 대조 결과 **우리 추정이 실측보다 3dB 낮게(더 조용하게) 나온다.**

```
신호 여유 = 그 선박의 RSSI − 잡음층
```

잡음층을 3dB 낮게 잡으면 여유를 3dB 크게 본다. `decode_margin = 10dB` 기준이 **실질 7dB로 관대하게** 작동했고, 환경성 유실(`RI_LOST_NOISE`)로 분류됐어야 할 것들이 원인 미상(`RI_LOST_PENDING`)으로 빠졌을 수 있다.

**할 일**
1. 프레임별로 추정치와 실측을 나란히 그리는 화면 (데이터 품질 페이지)
2. [logic.py](../dashboard/tabs/msg_analysis/logic.py) `_signal_margin`의 잡음층을 FSR 실측으로 교체
3. FSR이 없는 프레임은 추정치로 대체(fallback) — 어느 쪽을 썼는지 표시
4. 교체 전후로 유실 분류가 얼마나 달라지는지 비교

### B-2. 유실 분석에 프레임별 표 추가 ★

[page_loss.py](../dashboard/tabs/msg_analysis/page_loss.py)에 프레임 단위 표를 넣는다.

| 열 | 내용 | 출처 |
|---|---|---|
| 프레임 | 1분 단위 시각 (KST) | |
| 장소 / 구간 / 채널 | | |
| **FSR 슬롯수** | 장비가 정상 수신한 슬롯 수 | `ais_fsr.rx_slots` |
| **사용 슬롯** | 우리 로그의 VDM 파트 수 합 | 계산 |
| **못 받은 슬롯** | 위 둘의 차이 | 계산 |
| **CRC 실패** | 검출됐으나 디코딩 실패 | `ais_fsr.crc_fail` |
| **강신호 슬롯** | 잡음+10dB 이상 | `ais_fsr.strong_slots` |
| 잡음 | | `ais_fsr.noise_dbm` |
| 상태 | 정상 / 구간 시작·끝 / FSR 없음 | 계산 |

**"못 받은 슬롯"의 의미를 화면에 명시한다.** 이건 전파상 유실이 아니다. `rx_slots`는 "정상적으로 수신된" 슬롯이라 장비가 이미 디코딩에 성공한 것이다. 우리 로그에 없다면 **장비 출력과 파일 기록 사이의 손실**이다.

원문 기준 실측 (2,244개 프레임, 구간 시작·끝 제외):

| 못 받은 슬롯 | 프레임 수 | 비율 |
|---|---|---|
| 0개 | 336 | 15.0% |
| 1~2개 | 990 | 44.1% |
| 3~5개 | 521 | 23.2% |
| 6~10개 | 342 | 15.2% |
| 11개 이상 | 51 | 2.3% |
| 우리가 더 받음 | 4 | 0.2% |

1분에 400개 남짓 들어오는 중 보통 1~2개를 못 받는다. **로그 무결성 지표**로 쓴다 — 이 값이 갑자기 커지면 로거나 장비 문제다.

`rx_slots`는 **슬롯 수**라서 메시지 건수가 아니라 **VDM 파트 수**와 비교해야 한다. 메시지 건수로 세면 2,244개 프레임 중 맞는 게 하나도 없다(Type 5 같은 2파트 메시지가 슬롯 2개를 먹기 때문).

### B-3. 유실을 3단계로 나눈다

```
① 장비가 정상 디코딩        → rx_slots       (프레임당 약 440)
     └ 우리 로그에 기록됨     → 사용 슬롯       (1~2개 적음)  ← 기록 손실
② 장비가 검출했으나 실패     → crc_fail       (프레임당 약 48)  ← 진짜 유실
③ 장비가 검출조차 못 함      → 어디에도 없음                   ← 진짜 유실
```

**진짜 유실은 ②와 ③**이다. ①과의 차이(1~2개)와는 자릿수가 다르다.

- `crc_fail`은 검출된 신호의 **10.8%**. "신호는 왔다"는 증거라 **충돌 쪽**에 가깝다
- `crc_fail` 안에서 충돌인지 잡음인지는 이 값만으로 못 나눈다. `noise_dbm`이 높은 프레임의 `crc_fail`이 많은지 함께 본다
- 기존 `RI_LOST_NOISE` / `RI_LOST_PENDING`과 대응시켜, **유실 추정 건수와 `crc_fail`의 상관**을 확인한다

### B-4. 신호 여유가 부족한 메시지 특정

`strong_slots`는 개수만 주지 어느 메시지인지 알려주지 않는다. 메시지 단위로 직접 계산한다.

```
메시지별 신호 여유 = vsi_rssi − FSR.noise_dbm   (같은 장소·채널·프레임)
```

- **수신된 메시지** → 여유 10dB 미만인 것을 MMSI·슬롯까지 특정 가능
- **못 받은 것** → 개수만 추정

### B-5. `ext_res` — 슬롯 체인 로직의 독립 검증 (후순위)

Type 1의 `slot_timeout`으로 "다음 프레임에도 이 슬롯을 유지한다"를 아니까, 프레임 T−1 수신분으로 **T의 예약 슬롯 수를 예측**할 수 있다.

```
우리 예측       T-1 수신분 중 다음 프레임 슬롯을 예약한 국의 수
FSR.ext_res     수신기가 파악한 T 의 외부 예약 슬롯 수
차이            수신기는 아는데 우리는 모르는 국 → 우리가 놓친 양
```

수신기는 CRC 실패분까지 슬롯 점유로 세므로 우리보다 많이 안다. 편차가 커서(±25 수준) 마지막에 다룬다.

### B-6. 쓰지 않는 필드

`tx_slots`와 `own_res`는 **2,294건 전부 0**이다. 수신 전용국이라 당연하다. 저장은 하되 화면에는 내보내지 않는다.

## C. 장비 상태 표시

### C-1. 세 가지 상태

원문 확인 결과 장비 상태는 on/off 두 가지가 아니다.

| 상태 | AIVDM/VSI | FSR 등 상태문장 | 의미 | 처리 |
|---|---|---|---|---|
| **정상** | 있음 | 있음 | 완전 가동 | 전부 사용 |
| **반쪽 가동** | 있음 | **없음** | 재기동 직후 상태 출력 미복구 | 메시지 분석 ✅ / FSR 지표만 제외 |
| **중단** | 없음 | 없음 | 장비 꺼짐 · 장소 이동 | **구간 경계** |

**"FSR 없음 = 장비 꺼짐"은 틀렸다.** kmou KST 10:52~10:59 8분간 메시지가 분당 450~800건 정상 수신되는데 `$AIFSR`/`$PSTT`/`$AIALR`/`$AIADS`가 전부 0이었다. 장비는 받고 있었고 상태 출력 계통만 죽어 있었다.

그 구간 데이터 품질도 확인했다 — RSSI·SNR 분포와 슬롯 번호 범위가 정상 구간과 구분되지 않는다. **버릴 이유가 없다.**

그래서 **구간을 나누는 기준은 FSR이 아니라 메시지 스트림의 침묵**이다. 반쪽 가동 구간은 구간을 끊지 않고, 그 프레임의 FSR 지표만 비게 둔다(`LEFT JOIN`이 자동 처리).

원문 기준 FSR이 없는 프레임은 **28개**다.

### C-2. 토폴로지 화면

```

구간        ─1─│──── 2 ────────── │      │ ─3─│─4─│─5─│── 6 ──

FSR 유무    ━━━━━━━━━ ✕ ━━━━━━━━━ │      │ ━━━ ✕✕ ━━━━━━
```

## 추가로 검토할 것

**`$AIALR` / `$AIADS` / `$PSTT` 적재** — 지금 버리고 있다. `$AIALR`은 이름 그대로 **알람** 문장이라 "왜 반쪽만 켜졌는지"를 장비가 직접 말해줄 가능성이 크다. 내용을 먼저 확인한 뒤 적재 여부를 정한다.

**같은 파일명으로 내용만 교체하면 적재가 건너뛴다** — `src_file` 문자열만 비교하므로, 내용을 바꿔도 이름이 같으면 업데이트 모드가 조용히 넘어간다. 크기나 해시를 함께 저장하면 감지할 수 있다.

**`antenna_h_m`** — `rx_sites`에 컬럼만 있고 값이 없다. 거리–RSSI baseline의 주된 변수라 값을 알면 채운다. 채울 계획이 없으면 컬럼을 빼는 것도 방법.

**프레임을 컬럼으로 저장** — 프레임 계산(`date(recv_time-9h) + VSI 시:분 + 9h`, 자정 보정 포함)이 대시보드·검증 쿼리에 반복된다. `ais_msg_*`에 `frame` 컬럼을 추가하면 계산이 사라진다. `ais_fsr.frame`을 생성 컬럼으로 둔 것과 같은 발상. 스키마 변경이라 `--rebuild`가 필요하다.

| 1 | **A-1~A-4 구간 도입** + `LOGIC_VERSION` 4 | 나머지 전부의 전제. 오탐 1,369건 제거 |
